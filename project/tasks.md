## Backend

  - Acceptance Criteria: Проект запускается локально; есть health endpoint; конфигурация окружения читается из `.env`; структура каталогов соответствует архитектуре.
  - Estimate (часы): 6

- [ ] Спроектировать и применить схему PostgreSQL для операционных метаданных
  - Description: Создать миграции для таблиц `comparison_jobs`, `comparison_summaries`, `comparison_artifacts`, `comparison_events`.
  - Acceptance Criteria: Все таблицы создаются миграцией; есть индексы по `status`, `created_at`, `expires_at`, `job_id`; схема соответствует `architecture.md`.
  - Estimate (часы): 6

- [ ] Реализовать конфигурацию временного файлового хранилища
  - Description: Добавить сервис для сохранения загруженных DBF, промежуточных артефактов и готовых отчетов во временный storage с TTL-метаданными.
  - Acceptance Criteria: Файлы сохраняются в изолированный каталог; пути и сроки жизни сохраняются в БД; есть unit-тесты на создание и удаление артефактов.
  - Estimate (часы): 6

- [ ] Реализовать endpoint загрузки двух DBF-файлов
  - Description: Создать `POST /api/v1/comparisons/uploads` с приемом двух файлов, базовой валидацией расширения, размера и созданием `comparison_job`.
  - Acceptance Criteria: Endpoint принимает ровно два файла; при невалидных файлах возвращает понятную ошибку; при успехе создает запись задания и сохраняет файлы во временное хранилище.
  - Estimate (часы): 8

- [ ] Реализовать модуль предварительного чтения структуры DBF
  - Description: После загрузки считывать поля каждого DBF, определять кодировку `cp866` / `cp1251` и возвращать доступные поля для выбора ключей.
  - Acceptance Criteria: Для валидных DBF возвращаются списки полей и определенные кодировки; ошибки чтения корректно сериализуются в API; значения не нормализуются.
  - Estimate (часы): 12

- [ ] Реализовать endpoint запуска сравнения
  - Description: Создать `POST /api/v1/comparisons/{jobId}/run` с валидацией `key1`, `key2`, `structureOnly`, `checkFieldOrder` и постановкой задачи в очередь.
  - Acceptance Criteria: `key1` обязателен; `key2` опционален; нельзя запустить задание без ранее загруженных файлов; статус задания меняется на `queued`.
  - Estimate (часы): 6

- [ ] Реализовать worker и очередь фоновых задач
  - Description: Настроить Celery worker, сериализацию payload задания и обработку статусов `queued`, `processing`, `completed`, `failed`.
  - Acceptance Criteria: Worker забирает задачу из очереди; прогресс и финальный статус фиксируются в БД; при исключении задача помечается как `failed`.
  - Estimate (часы): 10

- [ ] Реализовать модуль строгого чтения DBF без нормализации значений
  - Description: Построить адаптер поверх `dbfread`, который отдает схему и строки в исходном виде без trim, normalization и скрытых преобразований.
  - Acceptance Criteria: Скрытые символы, пробелы и управляющие символы сохраняются в значениях; покрыты тестами edge-case примеры; логика не использует pandas как source of truth.
  - Estimate (часы): 14

- [ ] Реализовать модуль сравнения структуры
  - Description: Добавить логику выявления отсутствующих полей, лишних полей, несовпадения типов и отличий очередности полей.
  - Acceptance Criteria: Результат сравнения содержит отдельные коллекции для `STRUCTURE`, `TYPES`, `FIELD_ORDER`; поведение зависит от флага `checkFieldOrder`; есть unit-тесты.
  - Estimate (часы): 12

- [ ] Реализовать модуль анализа ключей и дубликатов
  - Description: Добавить построение одинарного и составного ключа, проверку существования ключевых полей и поиск дубликатов в обоих файлах.
  - Acceptance Criteria: Поддерживаются `key1` и `key1+key2`; дубликаты считаются отдельно по каждому файлу; ошибки отсутствующих ключей возвращаются как валидируемые бизнес-ошибки.

  - Acceptance Criteria: Формируются отдельные списки missing/extra rows; логика корректно работает для одинарного и составного ключа; результаты пригодны для UI preview и Excel.
  - Estimate (часы): 14

- [ ] Реализовать модуль сравнения значений полей без нормализации

## Milestones (спринты)

- Спринт 1: Технический фундамент
  - Backend: каркас FastAPI, схема БД, временное хранилище, upload endpoint, предварительное чтение структуры.
  - Frontend: каркас React/Vite, layout рабочего экрана, блок загрузки файлов, интеграция upload, выбор ключей.
  - DevOps / Infra: Dockerfiles, docker-compose, env-конфигурация.
  - QA: тестовая стратегия и первичный набор DBF-файлов.
  - Outcome: пользователь может загрузить два DBF и увидеть доступные поля для выбора ключей.

- Спринт 2: Ядро сравнения
  - Backend: worker, strict DBF reader, schema comparison, key analysis, reconciliation, mode `Только структура`, summary aggregation.
  - Frontend: параметры сравнения, action-panel, polling статуса, базовый экран результатов.
  - DevOps / Infra: reverse proxy, базовые health checks.
  - QA: приемочные сценарии для compare pipeline.
  - Outcome: пользователь может запускать сравнение и получать summary по структуре, ключам и данным.

- Спринт 3: Отчетность и детализация
  - Backend: сравнение значений без нормализации, preview endpoints, Excel generator, report download, cleanup endpoints.
  - Frontend: таблица категорий, preview различий, warning-блок strict comparison, скачивание отчета, обработка ошибок.
  - DevOps / Infra: CI pipeline, runtime monitoring, storage cleanup policy.
  - QA: функциональная валидация Excel и регрессия hidden-symbol сценариев.
  - Outcome: пользователь получает полный функциональный паритет основного сценария и скачивает детализированный Excel-отчет.

- Спринт 4: Стабилизация и pilot release
  - Backend: observability polishing, тесты, исправление дефектов производительности и memory hotspots.
  - Frontend: визуальный polishing, адаптивность, компонентная консолидация, тесты.
  - DevOps / Infra: runbook on-premise deployment, pilot configuration.
  - QA: smoke, pilot verification, финальный regression pass.
  - Outcome: система готова к внутреннему pilot release на сервере компании.
