# System Architecture

## High-Level Architecture
- Для данного продукта оптимальна архитектура `web client + modular backend + background job worker` с развёртыванием на внутреннем сервере компании.
- Архитектурный стиль: модульный монолит, а не микросервисы. Это соответствует узкому функциональному scope, внутреннему использованию, отсутствию внешних интеграций и необходимости быстро добиться функционального паритета с desktop-версией.
- Система состоит из трех основных runtime-границ:
- браузерный frontend для загрузки файлов, настройки сравнения, просмотра кратких результатов и скачивания Excel-отчета;
- backend API, принимающий файлы, валидирующий параметры, создающий задачи сравнения, выдающий результаты и управляемый lifecycle временных файлов;
- background worker, исполняющий тяжелые операции чтения DBF, строгого сравнения без нормализации и генерации Excel-отчета.
- Временные входные файлы, промежуточные результаты сравнения и готовый Excel-отчет хранятся только краткоживуще во внутреннем файловом хранилище сервера; постоянное хранение бизнес-данных не используется.
- Реляционная база данных хранит только операционные метаданные: задачи сравнения, статусы, параметры запуска, агрегированные результаты и ссылки на временные артефакты.
- Для on-premise deployment система должна запускаться как набор контейнеров или как несколько процессов на одном сервере: `frontend`, `api`, `worker`, `postgres`, `redis`, временное локальное файловое хранилище.

## Components
- `Web Frontend`
  Отвечает за одноэкранный рабочий интерфейс, загрузку двух DBF-файлов, выбор ключей, настройку флагов `Только структура` и `Проверка очередности полей`, отображение краткой сводки, preview различий и запуск скачивания отчета.
- `API Gateway / Application Server`
  Реализует HTTP API, принимает multipart-загрузки, валидирует входные параметры, создает задачи сравнения, отдает состояние обработки, агрегированные результаты и файл отчета. Также управляет очисткой временных артефактов по TTL.
- `Comparison Orchestrator`
  Внутренний модуль backend-а, отвечающий за orchestration пользовательского сценария: связывает загруженные файлы с задачей, проверяет обязательность `Ключа 1`, подготавливает job payload и публикует задачу в очередь.
- `DBF Reader Module`
  Модуль worker-а, отвечающий за открытие DBF-файлов, определение кодировки `cp866` / `cp1251`, чтение схемы и строк без нормализации значений.
- `Schema Comparison Module`
  Выявляет отсутствующие поля, лишние поля, несовпадения типов данных и различия в очередности полей.
- `Key Analysis Module`
  Проверяет наличие выбранных ключей, строит одинарный или составной ключ, проверяет уникальность, находит дубликаты и формирует техническую основу для reconciliation.
- `Data Reconciliation Module`
  Сопоставляет строки двух файлов по ключам, находит отсутствующие и лишние строки, сравнивает общие поля строго по исходным значениям без нормализации, включая скрытые символы.
- `Results Aggregation Module`
  Формирует краткую сводку для UI и структурированное представление результатов для Excel-отчета.
- `Excel Report Generator`
  Генерирует `.xlsx` с листами `SUMMARY`, `STRUCTURE`, `FIELD_ORDER`, `TYPES`, `DUPLICATES`, `RECONCILIATION`, `DETAILS` в формате, максимально близком к desktop-версии.
- `Job Queue`
  Отделяет пользовательский HTTP request lifecycle от тяжелой обработки больших DBF-файлов и генерации Excel.
- `Temporary File Storage`
  Хранит DBF-файлы, сериализованные промежуточные результаты и итоговый Excel-отчет только на период обработки и скачивания.
- `Operational Database`
  Хранит метаданные задач, параметры сравнения, статусы, агрегированные счетчики, ошибки и TTL для cleanup.
- `Cleanup Scheduler`
  Периодический внутренний процесс, удаляющий просроченные временные файлы и завершенные задачи старше заданного retention-периода.

## Data Flow
- Загрузка и запуск сравнения:
  Пользователь открывает web-интерфейс и загружает `Файл 1` и `Файл 2`. Frontend отправляет их в backend через multipart upload. Backend сохраняет файлы во временное хранилище, создает `comparison_job` в статусе `uploaded` и возвращает идентификатор задания.
- Предварительное чтение структуры:
  После загрузки backend либо сразу, либо через легковесную background-задачу читает заголовки DBF и список полей, чтобы frontend мог показать доступные ключи. Frontend получает список полей и активирует блок выбора `Ключа 1` и `Ключа 2`.
- Запуск полной проверки:
  Пользователь выбирает ключи и флаги сравнения, затем запускает проверку. Backend валидирует параметры, обновляет `comparison_job`, публикует задачу в очередь и переводит ее в статус `queued`.
- Выполнение сравнения:
  Worker забирает задачу, читает оба файла через `DBF Reader Module`, определяет кодировку, извлекает схему и строки. Затем последовательно выполняются модули сравнения структуры, анализа ключей и reconciliation данных.
- Формирование результатов:
  Worker агрегирует результаты в компактную структуру для UI и расширенную структуру для Excel. Эти данные сохраняются как метаданные и как временный serialized artifact.
- Генерация отчета:
  Если сравнение завершено успешно, worker генерирует `.xlsx`, сохраняет его во временное хранилище, а в базе фиксирует путь, checksum и статус `completed`.
- Просмотр результатов:
  Frontend опрашивает API по `job_id`, получает текущий статус, краткую сводку, preview категорий различий и сообщения об ошибках. Пользователь видит результаты в интерфейсе еще до скачивания отчета.
- Скачивание отчета:
  Пользователь запрашивает Excel-отчет. Backend проверяет статус задания, выдает файл из временного хранилища потоковой передачей и не сохраняет его дольше операционного TTL.
- Очистка:
  Cleanup scheduler удаляет DBF-файлы, сериализованные результаты и Excel-отчеты после истечения TTL. В базе остаются только минимальные технические метаданные на короткий срок либо очищаются полностью, если это согласовано эксплуатационно.

## API Design
- API-стиль: REST JSON API для метаданных и `multipart/form-data` для загрузки файлов. Такой формат проще для внутреннего web-продукта, чем GraphQL или gRPC, и лучше соответствует узкому набору экранов и действий.
- Аутентификация: на первом этапе отсутствует на уровне приложения, в соответствии с PRD. Доступ ограничивается корпоративной сетью, reverse proxy, VPN или внутренним контуром.
- Основные endpoints:
- `POST /api/v1/comparisons/uploads`
  Принимает два DBF-файла, создает задачу, сохраняет артефакты, возвращает `job_id` и список полей при успешном предварительном чтении.
- `POST /api/v1/comparisons/{jobId}/run`
  Принимает параметры запуска:
  `key1` обязательный;
  `key2` опциональный;
  `structureOnly` boolean;
  `checkFieldOrder` boolean.
  Валидирует параметры и ставит задачу в очередь.
- `GET /api/v1/comparisons/{jobId}`
  Возвращает статус задачи, параметры, ошибки, агрегированные метрики и readiness отчета.
- `GET /api/v1/comparisons/{jobId}/summary`
  Возвращает краткую сводку для экранов результатов.
- `GET /api/v1/comparisons/{jobId}/preview?section=DETAILS&limit=100&offset=0`
  Возвращает preview конкретной секции результата: `STRUCTURE`, `FIELD_ORDER`, `TYPES`, `DUPLICATES`, `RECONCILIATION`, `DETAILS`.
- `GET /api/v1/comparisons/{jobId}/report`
  Возвращает готовый Excel-файл по streaming download.
- `DELETE /api/v1/comparisons/{jobId}`
  Явно удаляет временные артефакты и запись задания раньше TTL; полезно для ручной очистки.
- Контракты ответов должны включать:
- `status`: `uploaded | ready_for_run | queued | processing | completed | failed | expired`;
- `warnings`: массив предупреждений, включая информацию о строгом сравнении без нормализации и возможных дубликатах ключей;
- `summary`: агрегированные счетчики;
- `report`: признак готовности и URL скачивания при наличии.

## Database Schema
- База данных нужна только для операционного состояния и не должна становиться постоянным хранилищем файлового контента.
- Таблица `comparison_jobs`
  `id` UUID PK;
  `status` varchar;
  `created_at` timestamp;
  `updated_at` timestamp;
  `expires_at` timestamp;
  `file1_original_name` varchar;
  `file2_original_name` varchar;
  `file1_temp_path` varchar;
  `file2_temp_path` varchar;
  `file1_size_bytes` bigint;
  `file2_size_bytes` bigint;
  `file1_encoding` varchar nullable;
  `file2_encoding` varchar nullable;
  `key1` varchar nullable;
  `key2` varchar nullable;
  `structure_only` boolean;
  `check_field_order` boolean;
  `error_code` varchar nullable;
  `error_message` text nullable;
  `report_temp_path` varchar nullable;
  `report_size_bytes` bigint nullable;
  `report_checksum` varchar nullable.
- Таблица `comparison_summaries`
  `job_id` UUID PK/FK;
  `file1_row_count` bigint;
  `file2_row_count` bigint;
  `common_field_count` integer;
  `missing_fields_count` integer;
  `extra_fields_count` integer;
  `type_mismatches_count` integer;
  `field_order_mismatches_count` integer;
  `duplicate_keys_count_file1` integer;
  `duplicate_keys_count_file2` integer;
  `missing_rows_count` bigint;
  `extra_rows_count` bigint;
  `data_differences_count` bigint;
  `has_differences` boolean.
- Таблица `comparison_artifacts`
  `id` UUID PK;
  `job_id` UUID FK;
  `artifact_type` varchar;
  `storage_path` varchar;
  `content_type` varchar;
  `size_bytes` bigint;
  `expires_at` timestamp.
- Таблица `comparison_events`
  `id` bigserial PK;
  `job_id` UUID FK;
  `event_type` varchar;
  `payload_json` jsonb;
  `created_at` timestamp.
- Детальные результаты по строкам и полям не рекомендуется полностью хранить в PostgreSQL для крупных файлов. Их лучше сериализовать во временный файл `json` / `msgpack` во внутреннем storage и отдавать preview порционно через worker-generated index.

## Tech Stack (обоснованный)
- `Frontend: React + TypeScript + Vite`
  Подходит для современного внутреннего интерфейса с насыщенным состоянием экрана, асинхронными статусами обработки и табличными preview. Это легче и быстрее в разработке, чем более тяжелые enterprise-framework решения.
- `UI: CSS variables + component library уровня Radix primitives или headless-подход`
  Соответствует `design.md`, где важны аккуратные внутренние инструменты, а не шаблонный внешний SaaS. Headless-компоненты дают контроль над аналитическим интерфейсом и статусными таблицами.
- `Backend API: Python + FastAPI`
  Наилучший компромисс под эту задачу, потому что экосистема Python хорошо подходит для работы с DBF, табличными данными и Excel, а FastAPI дает простой typed API для внутреннего продукта. Это снижает риск расхождения логики между API и worker по сравнению с разделением на разные языки.
- `Worker: Python`
  Тот же язык, что и backend, позволяет переиспользовать модули чтения DBF, сравнения и генерации отчетов без межъязыковых оберток.
- `DBF processing: dbfread + собственный слой адаптации`
  Извлечение из приложенного `.exe` показало использование `dbfread`; это повышает шанс приблизить web-логику к desktop-поведению. Поверх библиотеки нужен собственный strict-comparison слой без нормализации.
- `Data processing: pandas только там, где это помогает отчетности, но не как единственный источник truth`
  Для критического строгого сравнения значений лучше собственная детерминированная логика построчного сравнения, чтобы не допустить неявных преобразований типов и значений.
- `Excel generation: XlsxWriter`
  Также обнаружен в составе desktop-приложения; это хороший выбор для воспроизводимого `.xlsx` с несколькими листами и контролируемым форматированием.
- `Database: PostgreSQL`
  Надежно хранит операционные метаданные, статусы и агрегаты; избыточная аналитическая СУБД не требуется.
- `Queue / cache: Redis`
  Простое и достаточное решение для фоновых задач и быстрых статусов. Подходит лучше RabbitMQ или Kafka, так как нагрузки и интеграционный ландшафт пока ограничены.
- `Background jobs: Celery` или `RQ`, предпочтительно `Celery`
  `Celery` лучше подходит, если потребуется контроль retries, progress states и разделение тяжелых очередей по типам задач.
- `Deployment: Docker Compose для pilot / внутреннего внедрения`
  Это соответствует вероятному on-premise сценарию и упрощает внутреннее развёртывание без Kubernetes на первом этапе.

## Integrations
- Внешние бизнес-интеграции по PRD не требуются.
- Технически система интегрируется только с:
- файловой системой сервера или внутренним mounted volume для временных файлов;
- PostgreSQL для метаданных;
- Redis для очередей и статусов.
- Reverse proxy уровня `Nginx` или `Traefik` рекомендуется как инфраструктурная обвязка для внутренних TLS-терминации, ограничения размера загрузки и маршрутизации до frontend / API, хотя это не является продуктовой интеграцией.

## Scalability
- Основная нагрузка определяется не количеством пользователей, а размером DBF-файлов и стоимостью сравнения строк и генерации Excel.
- Модульный монолит масштабируется вертикально на раннем этапе: больше CPU, RAM и быстрый SSD под временные файлы.
- Worker-контур масштабируется независимо от frontend/API: можно запускать несколько worker-процессов, ограничивая параллельность, чтобы не перегружать сервер при одновременных тяжелых сравнениях.
- Очередь задач защищает API от долгих blocking-операций и позволяет возвращать пользователю статус `queued` / `processing`.
- Preview-результаты должны отдаваться пагинированно, чтобы не загружать browser и API очень большими ответами.
- Для крупных файлов стоит использовать потоковое чтение DBF там, где это возможно, и хранить промежуточные индексы на диске, а не удерживать весь набор данных в памяти без необходимости.
- Bottleneck-ы:
- построение ключевых индексов для больших файлов;
- сравнение большого количества полей в `DETAILS`;
- генерация тяжелого Excel-отчета.
- Если later-stage нагрузка вырастет, первым шагом масштабирования должен быть вынос worker pool на отдельные узлы, а не дробление системы на микросервисы.

## Security
- Поскольку авторизация на уровне приложения не требуется, основной контроль доступа переносится на сетевой периметр: внутренний контур компании, reverse proxy, firewall, VPN и сегментация сети.
- Все загруженные DBF-файлы и сформированные отчеты считаются чувствительными внутренними данными и не должны храниться дольше, чем нужно для завершения сценария.
- Временные файлы должны храниться в каталоге с ограниченными правами доступа и удаляться по TTL и по явной команде очистки.
- Backend должен валидировать MIME/type, расширение, базовую структуру DBF и лимиты размера файла до передачи в worker.
- Необходимо исключить логирование содержимого строк DBF и значений полей. В логи допускаются только технические метаданные, счетчики и коды ошибок.
- Секреты инфраструктуры должны храниться во внутренних переменных окружения или secret-store, не в репозитории.
- Скачивание отчета должно быть привязано к действующему `job_id` и разрешено только до истечения TTL артефакта.
- При ошибках интерфейс и API не должны возвращать чувствительные внутренние пути файловой системы или stack traces.

## Risks
- Главный архитектурный риск: отсутствие эталонных DBF-файлов и готовых Excel-отчетов усложняет доказательство полного функционального паритета с desktop-версией.
- Обнаруженные в PRD требования к строгому сравнению без нормализации означают, что даже удобные табличные библиотеки нельзя использовать без осторожного обрамления, иначе возможны скрытые преобразования значений.
- Не до конца определена инфраструктурная модель on-premise deployment: один сервер, VM-кластер или контейнерная платформа. Это влияет на HA, storage strategy и операционные процедуры.
- При крупных файлах есть риск высоких требований к памяти и времени обработки, особенно при формировании листа `DETAILS`.
- Отсутствие авторизации упрощает продукт, но делает качество сетевого периметра и инфраструктурного доступа критически важным.
- Если later-stage потребуется история запусков, аудит или многопользовательская координация, текущая минималистичная схема хранения потребует расширения.
