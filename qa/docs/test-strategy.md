# Sprint 1 QA Strategy

## Scope

Sprint 1 покрывает:
- загрузку двух DBF-файлов;
- предварительное чтение структуры;
- определение кодировки `cp866` / `cp1251`;
- получение списка полей для выбора ключей;
- базовую валидацию файлов и ошибок чтения.

## Test Levels

- Unit:
  Проверка временного файлового хранилища, DBF preview service, валидации upload endpoint, преобразования response payload.
- Integration:
  Проверка полного backend-потока `upload -> temp storage -> DBF preview -> DB write -> API response`.
- UI:
  Проверка выбора двух файлов, загрузки, отображения кодировки, количества полей и активации селектов ключей.
- Smoke:
  Проверка, что `docker compose up --build` поднимает `frontend`, `api`, `worker`, `postgres`, `redis`, `nginx`.

## Priority Scenarios

- Успешная загрузка двух валидных DBF с одинаковым списком общих полей.
- Успешная загрузка двух валидных DBF с разным составом полей.
- Ошибка при загрузке не-DBF файла.
- Ошибка чтения поврежденного DBF.
- Успешное определение `cp866`.
- Успешное определение `cp1251`.
- Возврат списка полей без нормализации названий.

## Risks To Track

- Поведение `dbfread` на нестандартных DBF-файлах.
- Различия между фактической кодировкой и автоопределением через fallback `cp866 -> cp1251`.
- Ошибки при крупных файлах уже на этапе предварительного чтения структуры.
- Возможные расхождения между локальным и docker-окружением.

## Exit Criteria For Sprint 1

- API upload endpoint стабильно принимает два валидных DBF-файла.
- Для валидных файлов возвращаются `job_id`, `status`, список полей и кодировки.
- UI отображает результат загрузки и активирует выбор ключей.
- Невалидные файлы не создают успешную задачу и возвращают понятную ошибку.
- Базовое pilot-окружение поднимается через Docker Compose.
